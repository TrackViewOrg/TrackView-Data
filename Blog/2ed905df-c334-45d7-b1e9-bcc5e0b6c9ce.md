The wait is finally over. After six months of re-engineering our core data ingestion pipeline, we are officially moving *ApexView Telemetry* into **Closed Beta Stage 3**.

## âš™ï¸ The "Slipstream" Data Engine

The biggest bottleneck in v2.x was handling the sheer volume of data during endurance events. A 24-hour race with 60Hz sample rates on 40+ channels created massive log files that were slow to parse.

We have deprecated the old JSON-based logging system in favor of a custom binary format powered by Rust.

### The New Data Packet Structure

We need testers to verify data integrity during high-load scenarios (e.g., multi-class racing with 50+ cars on track). Below is a snippet of the raw C++ struct representation of the new suspension telemetry packet. Note the move to `int16_t` for compressed damper velocity to save bandwidth.

```cpp
// ApexView v3.0 UDP Packet - Suspension Telemetry Block (Draft)
struct SuspensionDataPacket {
    uint32_t frameIdentifier;  // Syncs with graphics engine frame
    double timestampMS;         // High-precision race time

    // Per-wheel measurements (FL, FR, RL, RR)
    float suspensionTravelNormalized[4]; // 0.0 (bottomed) to 1.0 (full droop)
    int16_t damperVelocityCompression[4]; // Compressed mm/s
    int16_t damperVelocityRebound[4];     // Compressed mm/s
    
    float rideHeight[4];                 // Measured in meters from reference plane
    
    // New for v3.0: Bump Stop engagement Boolean flag packed into a byte
    uint8_t bumpStopActiveFlags;         // Bitmask: 0x1=FL, 0x2=FR, 0x4=RL, 0x8=RR

    void Serialize(Buffer& buffer) {
        // Implementation hidden for brevity; uses Little Endian
        buffer.write(frameIdentifier);
        buffer.write(timestampMS);
        buffer.writeArray(suspensionTravelNormalized, 4);
        // ... rest of serialization
    }
};

```

If you see "spikes" in your damper velocity graphs where the value suddenly jumps to `32767` (the max int16 value), please export the raw CSV log and submit it via Jira ticket #T-994.

## ğŸ“‹ The Testing Protocol: How to Break It

We don't want you to just drive laps. We want you to try and crash the application.

### Scenario A: The Endurance Stress Test

1. Start a 4-hour endurance session in ACC or iRacing.
2. Set the telemetry logging rate to "Max / Raw".
3. Keep the live telemetry dashboard open on a second monitor the entire time.
4. **Goal:** See if the RAM usage "leaks" over time. It should stay stable around 300MB during active logging.

### Scenario B: The Alt-Tab Torture Test

1. Start logging data in full-screen mode.
2. Rapidly Alt-Tab out of the sim and into the Telemetry app, resize the window immediately, and Alt-Tab back into the sim.
3. Repeat this 20 times rapidly.
4. **Goal:** Try to trigger a rendering thread crash or a desync between the live data and the graphs.
